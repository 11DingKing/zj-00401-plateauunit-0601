## 四、机组从建档到并网的完整状态流转

### 4.1 阶段 1：建档（create_unit）

入口：`POST /api/units` → [services.py create_unit()](file:///Users/ding/Documents/SOLOCODE%203/0619/macmini/zj-00401-plateauunit-5/app/services.py#L273-L305)

**发生了什么：**
1. 扫风面积若未填，按 `π × (叶轮直径/2)²` 自动计算 → 调 `_compute_swept_area()`
2. 创建 Unit 记录
3. **关键：自动初始化 5 道 Gate，全部 `status=pending`，按 `GATE_ORDER` 顺序写 `stage_index`**
   ```python
   for idx, stage in enumerate(GATE_ORDER):
       unit.gates.append(Gate(stage=stage.value, stage_index=idx, status=GateStatus.pending.value))
   ```
4. 此时状态：所有关 pending，current_stage=foundation（基础验收），operational=False

### 4.2 阶段 2：逐步通关（update_gate_status）

入口：`POST /api/units/{unit_id}/gates/{stage}` → [services.py update_gate_status()](file:///Users/ding/Documents/SOLOCODE%203/0619/macmini/zj-00401-plateauunit-5/app/services.py#L390-L518)

这是**整个系统最核心的函数**，三类状态转换对应三条不同的逻辑分支：

#### 分支 A：目标状态 = passed（通关）
按顺序执行 4 条**硬校验**，任意一条失败都会抛 HTTPException：

| 序号 | 校验规则 | 代码位置 | 拦截含义 |
|------|----------|----------|----------|
| 1 | **上一关必须已 passed** | [L415-L422](file:///Users/ding/Documents/SOLOCODE%203/0619/macmini/zj-00401-plateauunit-5/app/services.py#L415-L422) | 保证必须按 `GATE_ORDER` 顺序推进 |
| 2 | **该关卡下不得有未关闭问题** | [L424-L431](file:///Users/ding/Documents/SOLOCODE%203/0619/macmini/zj-00401-plateauunit-5/app/services.py#L424-L431) | `_open_issues_for_stage()` 查 stage 相同且 status=open 的问题 |
| 3 | **进入空载调试（第③关）及以后，4 类专项配置必须全部启用** | [L433-L444](file:///Users/ding/Documents/SOLOCODE%203/0619/macmini/zj-00401-plateauunit-5/app/services.py#L433-L444) | `_missing_config_types()` 检查 enabled=True 的类型是否覆盖了所有 ConfigType |
| 4 | **并网确认（第⑤关）时，机组不能有任何逾期问题** | [L446-L453](file:///Users/ding/Documents/SOLOCODE%203/0619/macmini/zj-00401-plateauunit-5/app/services.py#L446-L453) | `_grid_blocked_by_overdue()` 遍历所有 issue，有 open 且 due_date<now 就拦截 |

全部通过后：写入 `started_at`（首次执行时）、`completed_at=now`、`status=passed`。

#### 分支 B：目标状态 = failed（未通过）
1. 写入 `started_at`、`completed_at=now`、`status=failed`
2. **关键副作用：后续所有关卡回退为 pending**（清除 completed_at/operator/remarks），保证"失败关卡之后的关卡不可先行"不变式。见 [L469-L473](file:///Users/ding/Documents/SOLOCODE%203/0619/macmini/zj-00401-plateauunit-5/app/services.py#L469-L473)
3. 若请求带了 `issue_title`，自动创建一条关联该 stage 的 Issue（支持顺便指派 team/due_date）

#### 分支 C：目标状态 = pending（重置）
将该关及后续所有关全部重置为 pending（清空时间/执行人/备注）。

### 4.3 阶段 3：并网确认通过（已投运）

判定函数：[_is_operational()](file:///Users/ding/Documents/SOLOCODE%203/0619/macmini/zj-00401-plateauunit-5/app/services.py#L52-L54)
```python
def _is_operational(unit: Unit) -> bool:
    g = _gate_by_stage(unit, GateStage.grid_connection)
    return bool(g and g.status == GateStatus.passed.value)
```
**含义：只要 grid_connection 这一关通过了，就算 operational（已投运）。**

"当前应执行关卡"判定：[_current_stage()](file:///Users/ding/Documents/SOLOCODE%203/0619/macmini/zj-00401-plateauunit-5/app/services.py#L57-L63) → 遍历 `GATE_ORDER`，返回第一个非 passed 的关卡；全部通过返回 None。

## 五、专项配置（SpecialConfig）的作用

### 5.1 生命周期
- 创建/更新：`POST /api/units/{unit_id}/configs` → [upsert_config()](file:///Users/ding/Documents/SOLOCODE%203/0619/macmini/zj-00401-plateauunit-5/app/services.py#L340-L371)
  - 同一 unit_id + config_type 不存在就 insert，存在就 update（upsert 语义）
  - `enabled=True` 才算"已启用"，这是覆盖率统计的唯一口径

### 5.2 "配置补齐"的判定口径

两个辅助函数贯穿整个系统，**口径完全一致**：
- [_missing_config_types(unit)](file:///Users/ding/Documents/SOLOCODE%203/0619/macmini/zj-00401-plateauunit-5/app/services.py#L188-L191)：返回尚未启用的配置类型列表
- [_config_coverage_complete(unit)](file:///Users/ding/Documents/SOLOCODE%203/0619/macmini/zj-00401-plateauunit-5/app/services.py#L194-L196)：返回布尔值（是否全部启用）

口径：遍历 `ConfigType` 枚举的全部 4 个值，每个值都必须存在对应的 SpecialConfig 且 `enabled=True`，缺一即为"未补齐"。

### 5.3 配置对验收流程的影响（唯一一处硬拦截）
在 `update_gate_status()` 校验点 3：**关卡序号 ≥ no_load_debug（即第③关及以后）时，必须已补齐全部 4 类配置**，否则不让 passed。

→ 基础验收、吊装验收阶段可以先不补配置，但到空载调试必须全部到位。

## 七、统计汇总的数据来源详解

### 7.1 全局统计：compute_stats()

定义在 [services.py L665-L848](file:///Users/ding/Documents/SOLOCODE%203/0619/macmini/zj-00401-plateauunit-5/app/services.py#L665-L848)，各字段的数据来源：

| 统计字段 | 计算方式 | 数据来源表 |
|----------|----------|-----------|
| total_units | `len(db.query(Unit).all())` | units |
| operational_count | 筛选 `_is_operational(u)==True` 的机组数 | units + gates(grid_connection.passed) |
| first_batch_operational_count | operational 且 grid_batch=="首批" | units |
| average_debug_hours | 所有已投运机组的 `(并网.completed_at - 基础.started_at)` 取均值（小时） | gates |
| unclosed_issue_count | 所有 status=open 的 issue 数 | issues |
| overdue_unclosed_count | 所有 open 且 overdue==True 的 issue 数 | issues |
| config_coverage[].coverage | 按 ConfigType 分组：enabled=True 的机组数 / 总机组数 | special_configs + units |
| config_complete_count | `_config_coverage_complete(u)==True` 的机组数 | special_configs（enabled=True 覆盖全部 4 类） |
| config_incomplete_units | 未补齐机组的 missing_config_types 和 current_stage 明细 | 同上 + gates |
| unclosed_by_config | **按机组启用的配置类型"复制分发"到各桶**：一台启用了 3 类配置的机组，其 open 问题会被同时算进这 3 类；一台都没启用的归入"无专项配置"桶 | special_configs + issues |
| unclosed_by_slope | 按 unit.slope_position 分组的 open 问题数 / 逾期数 | units + issues |
| unclosed_by_team | 按 issue.team 分组，按 Team 枚举顺序输出，未分派单独一组 | issues |

### 7.2 坡位适配复盘：compute_slope_review()

定义在 [services.py L959-L1049](file:///Users/ding/Documents/SOLOCODE%203/0619/macmini/zj-00401-plateauunit-5/app/services.py#L959-L1049)，这是**统计维度最复杂**的一块：

1. **第一层分组**：按 `Unit.slope_position` 分组（山脊鞍部、东坡上段、北脊中段...）
2. **第二层分组**：每个坡位内，再按机组启用的专项配置类型拆子组（一台启用 2 类配置的机组会同时归入 2 个子组 + "无配置"组）
3. **每个子组输出 6 项指标**：
   - `acceptance_rate`：子组内 operational 机组占比（验收通过率）
   - `rework_count`：子组内所有机组的 issue 总数（含已关闭，作为"返工次数"）
   - `avg_rework_per_unit`：rework_count / unit_count
   - `avg_debug_hours`：子组内已投运机组的调试耗时均值
   - `stage_pass_rates`：5 道关各自的通过率（子组内该关 passed 的机组数 / 子组总数）
4. **风险评分**：[_compute_risk_score()](file:///Users/ding/Documents/SOLOCODE%203/0619/macmini/zj-00401-plateauunit-5/app/services.py#L887-L907)
   ```
   风险分 = (1 - 整体验收通过率) × 40   ← 通过率越低风险越高
         + min(平均返工/5, 1) × 30     ← 返工≥5次封顶
         + min(平均耗时/720h, 1) × 30  ← 30天封顶，无数据记满分30
   ```
   → 最后按风险分降序排列，高风险坡位排在最前面。
